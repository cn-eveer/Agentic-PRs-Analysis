# Codebook: Manual Annotation of Agentic Pull Requests

This document defines the manual coding scheme used in the study
"Why Are Agentic Pull Requests Merged or Rejected? An Empirical Study".

The goal of this codebook is to make the manual annotation process
transparent, reproducible, and interpretable by external researchers.

(Updated: Jan 25th 2026)


## 1. Coding Dimensions Overview

Manual coding is performed along two dimensions:

- **RQ1 (Rejected PRs): Decision Rationale**
  - Agentic Failure (AF)
  - Non-Agentic Failure (Non-AF)
  - Unknown

- **RQ2 (Merged PRs): Human Involvement Pattern**
  - Feedback Loop (FL)
  - Human Intervention (HI)
  - No Feedback Loop (No-FL)
  - Unknown

Each PR receives **exactly one label** within the applicable dimension.

---

## 2. Code Scheme for Rejected PRs (RQ1)

### 2.1 Agentic Failure (AF)

**Definition**
The PR is rejected because the agent’s change is incorrect or harmful. The code does not work as intended, breaks tests, fails checks, or is explicitly identified as wrong by reviewers.

**Typical Evidence Examples:**
- CI or tests fail and maintainers request fixes or close the PR (e.g., “Failing tests”).
- Reviewer states the change does not work or introduces a regression.
- Tooling signals like lint, build, or quality gates fail due to the submitted patch.

**Representative PR Examples (URLs):**
- https://github.com/robinkarlberg/transfer.zip-web/pull/34
- https://github.com/wandb/weave/pull/4719
- https://github.com/OpenHFT/Chronicle-Core/pull/701

---

### 2.2 Non-Agentic Failure (Non-AF)

**Definition**
The PR is rejected for reasons that do not indicate the agent’s code is broken. This includes workflow and process outcomes such as duplicates, superseded changes, missing context, inactivity, or reviewer decisions that reject otherwise plausible output.

**Typical Evidence Examples:**
- Closed as duplicate, superseded, or redirected to another PR or issue.
- Closed due to inactivity or stale branch policies without technical critique.
- Maintainer declines for scope, preference, or project context rather than correctness.

**Representative PR Examples (URLs):**
- https://github.com/joamag/boytacean/pull/24
- https://github.com/andrewlock/NetEscapades.EnumGenerators/pull/156
- https://github.com/airbytehq/airbyte/pull/55201

---

### 2.3 Unknown (Rejected PRs)

**Definition**
The PR is rejected, but the available artifacts do not provide enough observable evidence to determine whether the rejection was caused by agentic failure or non-agentic factors.

**Typical Evidence Examples:**
- PR is closed without comments or without actionable reviewer feedback.
- Review discussion is missing, inaccessible, or deleted.
- External links are unavailable and no decision rationale is recorded.

**Representative PR Examples (URLs):**
- https://github.com/hyperlight-dev/hyperlight/pull/513
- https://github.com/dotnet/aspire/pull/9653

---

## 3. Code Scheme for Merged PRs (RQ2)

### 3.1 Feedback Loop (FL)

**Definition**
Human reviewers provide comments, the agent revises the PR in response, and the final merged changes contain no human-authored code changes.

**Typical Evidence Examples:**
- Reviewer requests modifications, then agent pushes follow-up commits addressing them.
- Merge occurs after agent revisions and the diff history shows only agent commits.

**Representative PR Examples (URLs):**
- https://github.com/dotnet/aspnetcore/pull/62056
- https://github.com/airbytehq/airbyte/pull/63320

---

### 3.2 Human Intervention (HI)

**Definition**
Human reviewers provide comments and the agent revises the PR, but humans also contribute code changes in the PR before merge.

**Typical Evidence Examples:**
- Agent responds to review, and a human additionally edits code, fixes tests, resolves conflicts, or performs refactoring commits.
- Commit history includes both agent and human code changes.

**Representative PR Examples (URLs):**
- https://github.com/primer/view_components/pull/3512
- https://github.com/antiwork/helper/pull/405

---

### 3.3 No Feedback Loop (No-FL)

**Definition**
The PR is merged without an agent revision that responds to human review comments. The PR may be merged with no comments or with comments that do not lead to agent-authored corrective commits.

**Typical Evidence Examples:**
- Merged directly after submission with no review cycle requiring changes.
- Comments exist, but the agent does not revise the code based on them before merge.

**Representative PR Examples (URLs):**
- https://github.com/oven-sh/bun/pull/19827
- https://github.com/KittyCAD/modeling-app/pull/7035

---

### 3.4 Unknown (Merged PRs)

**Definition**
Merged PRs where available artifacts do not support a confident assignment to FL, HI, or No-FL.

**Typical Evidence Examples:**
- Comments are generated by an agent or bot, not a human reviewer.
- Human actions are unrelated to feedback-driven improvement, such as merging main into the branch.
- Links are unreachable or commit authorship is unclear.

**Representative PR Examples (URLs):**
- https://github.com/crewAIInc/crewAI/pull/1923
- https://github.com/microsoft/openvmm/pull/1690

## 4. Annotation Procedure

- Four annotators participate in the study.
- Annotator is grouped for each type of PR(accepted/rejected)

### 4.1 Inter-rater reliability is assessed using Cohen’s κ.
- Each annotator receives different sets of PRs to label with sample of 50 PRs.
- Annotated labels are use to calculate inter-rater agreement statistics.
- If annotators disagree on a label, they discuss to reach consensus.
- This process is repeated until the inter-rater reliability is satisfactory.

### 4.2 Main Annotation Process
- PRs are randomly assigend to annotators after the inter-rater reliability is established.
- Each PR is independently annotated by **two annotators**.
- Annotators assign labels based on this codebook.
- Disagreements are resolved through discussion.
- If consensus cannot be reached, the PR is labeled as `Unknown`.

## 5. Annotator Assignment

- PRs are stratified by agent and repository.
- Each PR is randomly assigned to two annotators.
